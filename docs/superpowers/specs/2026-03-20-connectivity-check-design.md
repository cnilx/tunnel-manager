# 守护进程连通性检测设计文档

日期：2026-03-20

## 背景

当前守护监控逻辑（`tunnel_gui.py` `monitor_tunnels()`）每 10 秒检查一次 SSH 进程是否存活。但进程存活不等于隧道可用——SSH 进程可能已挂起或连接已断开，此时本地端口不可达，但 PID 仍然存在，监控不会触发重连。

## 目标

在守护进程检测到进程存活的情况下，额外验证隧道是否真正联通，若不通则自动重连。

## 方案选择

选择**方案 B：在监控循环中内联一次快速 socket 探测**。

理由：
- 改动最小，逻辑集中在 `monitor_tunnels()` 一处
- 单次探测（1.5 秒超时），不阻塞整个监控循环
- 与现有代码风格完全一致，无需新增线程或状态

## 设计

### 修改位置

`source/tunnel_gui.py`，`monitor_tunnels()` 方法，第 1047-1090 行。

### 逻辑变更

**当前流程：**
1. 检查 PID 对应进程是否存活
2. 不存活 → 重启

**新流程：**
1. 检查 PID 对应进程是否存活
2. 不存活 → 重启（保持不变）
3. 存活 → 额外做一次 socket 连接探测本地端口（超时 1.5 秒，单次，无重试）
4. 探测失败 → 日志记录 → terminate 旧进程（kill fallback）→ 重启

### 连通性探测规格

- 协议：TCP socket connect
- 地址：`127.0.0.1:{local_port}`
- 超时：1.5 秒
- 重试次数：0（单次，失败即重连）
- 失败阈值：1 次失败即触发重连

### 重连规格

- 先 `terminate()` 旧进程，等待最多 3 秒
- 超时则 `kill()`
- 再调用 `self.manager.start_tunnel(tunnel)` 重启
- 更新 PID、刷新 UI 状态，与现有重启逻辑一致

### 日志消息

| 事件 | 日志内容 |
|------|----------|
| 探测失败触发重连 | `隧道 {name} 进程存活但不通，正在重连...` |
| 重连成功 | `隧道 {name} 已重连 (PID: {pid})` |
| 重连失败 | `隧道 {name} 重连失败` |

## 约束

- socket 探测仅在进程存活分支执行，不影响进程不存活分支
- 探测总耗时（1.5 秒）远低于监控间隔（10 秒），不会造成循环阻塞
- 不引入新线程、新配置项、新状态字段
